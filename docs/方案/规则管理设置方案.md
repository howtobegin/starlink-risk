# 规则管理设置方案

新增规则时，需要先选择模型。
模型一，只支持周期条件，事件属性匹配符号只支持 >、<、>=、<=、==、!=，不支持跨历史。

规则状态变更为'已上线'时将规则数据写入rule_json表中，规则状态变更为'已下线'时将规则数据从rule_json中删除。
等于说只有在上下线时才对rule_json表进行操作，其他编辑、审核等操作不要影响rule_json表，且只有rule_json与flink进行cdc对接。

如果一个规则有多个条件，那么配置的预警信息模板中使用kafkaEvent中的字段必须是多个条件都共有的，否则可能出现kafkaEvent字段的占位符得不到对应的数据的解析。

## 关于修改业务平台kafka推送数据事件及事件属性 注意事项

1. 无论是新增、删除、修改[业务平台kafka推送数据事件及事件属性]都需要先停止使用了属性的规则，因为规则数据真正修改是在重新上线的时候，不先停后上线修改了也是无用的，会导致误解。
2. 无论是新增、删除、修改[业务平台kafka推送数据事件及事件属性]
   ，除了需要先停止使用了属性的规则，还要让对应业务平台停止推送对应的事件数据。等修改事件属性完成后，才能重新上线规则，最后才能重新开始推送kafka数据。

## 规则目标

目标编号不让修改，事件编号、属性编号可以修改。

前端效验同一目标下事件编号不能重复，同一事件下属性编号不能重复。

## 关于下线规则运算机清理旧状态

记录每个产生状态的 规则编号、规则版本、目标字段、目标值，然后flink内部先使用布隆过滤器进行初步的去重，然后存入doris唯一表中供后续下线规则运算机使用。
下线规则运算机时，从doris查询获取到对应规则编号、规则版本的数据，然后cdc到flink，flink根据这些值进行目标字段、目标值的keyBy，然后clear旧状态值。
